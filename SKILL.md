# Ruishu Pure Calculation

## Core Principle

Build a browser-free collector by separating the stable Ruishu algorithm from the site-specific profile. Do not guess inner VM semantics when a real passing browser sample can be decoded and compared byte by byte.

Use this skill only for systems the user is authorized to test or integrate.

## Decision Tree

1. Request the target entry URL.
2. If it is not `412`, first rule out redirects, cached trust cookies, or a non-Ruishu path.
3. If it is `412` with `$_ts.cd`, `$_ts.nsd`, and an `xxxS` cookie, solve Cookie T first.
4. If the second entry request with `S+T` is not 200, stay in Cookie/basearr diagnostics.
5. If entry is 200 and the business API is 400, solve Ajax suffix/signature next.
6. If both entry and business API return expected data, document the profile and verification commands.

## Stage Gates

- Keep `cd`, `nsd`, JS, `S cookie`, and decoded samples from the same session bundle.
- Prove the stable encryption layer with a hybrid check: browser/sdenv basearr + pure encryption -> entry 200.
- Do not edit Ajax suffix code until entry-page second hop is 200.
- Do not call a target "pure solved" until a fresh session passes entry 200 without a browser.
- For business collectors, require a fresh-session API JSON check, not only a copied curl replay.

## Data-Driven Adaptation

For a new target, collect 3-5 passing browser sessions and decode their Cookie T values. Parse each `basearr` as TLV: `type, length, payload`. Compare Python-generated bytes against browser bytes with `scripts/compare_basearr.py`, then label differences as fixed, key-derived, time-derived, random, environment-derived, computed, or unknown.

Put site differences into a profile instead of hardcoding them into generic algorithms. Avoid variable-name hooks: Ruishu variable names are seeded by `nsd`, so durable anchors are structural patterns, constants, byte lengths, AST shape, and TLV positions.

## Parameter Classes

- Stable engine defaults: PRNG, key extraction shape, Cookie name expression, AES/CRC/Huffman/Base64 packaging. Verify with `references/constants.md`.
- Profile data: `type3`, `type10`, `type6`, `type2`, `type7`, `type9/type13`, codeUid JS choice, and Ajax suffix layout.
- Fresh session data: `cd`, `nsd`, `S`, `T`, all 45 `keys`, `JSESSIONID`, `r2t`, current time, `r20`, `nd`, and suffix values.
- Machine-sensitive data: UA, platform, viewport, language, timezone, system clock, proxy, and TLS behavior.

## Deliverables

For a completed target, deliver a browser-free Python collector, a site profile or documented constants, entry-page 200 verification, API JSON verification when in scope, and a short 400/412 failure guide.

## Common Mistakes

| Mistake | Fix |
| --- | --- |
| Reusing a captured `xxxT` or Ajax suffix | Generate fresh values from the current 412 challenge. |
| Treating one basearr length as universal | Branch by decoded browser samples and keys. |
| Guessing inner VM meaning | Decode passing Cookie T and compare bytes. |
| Mixing `cd` from one response with JS or cookies from another | Keep challenge data in one session bundle. |
| Hardcoding real-machine browser values | Move them into a profile and keep request headers consistent. |
| Seeing API 400 and changing Cookie T blindly | First prove entry-page second hop is 200. |
| Hooking by variable names | Hook by constants, AST shape, code length, and TLV structure. |
